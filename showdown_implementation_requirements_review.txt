Showdown 구현요구서 논리 검토 및 보완사항 v0.2

대상 문서
showdown_implementation_requirements.txt

목적
기존 구현요구서를 실제 코드와 스키마에 다시 대조하여 과대평가된 구현 상태, 요구사항 사이의 충돌, 빠진 정책과 추가 테스트 항목을 정리한다.


1. 반드시 수정할 논리 오류

1.1 CRUD 구현 수준
기존 문서의 관리자 대회·부문·선수 CRUD 완료 표현은 부분 완료로 수정한다.
- 부문 수정 API는 있지만 관리자 화면에 수정 기능이 없다.
- 대회 삭제 API는 있지만 관리자 화면에 삭제 기능이 없다.
- 경기 요청의 courtName과 refereeName은 DTO에 있으나 서비스에서 저장하지 않는다.
- 단계·조·심판·경기 CRUD도 관계 검증과 실제 배정 관계가 빠져 있으므로 부분 완료로 분류한다.

1.2 역할 체계
현재 역할 이름이 계층마다 다르다.
- Spring Security: ADMIN, REFEREE, PLAYER
- SQL 설계: system_admin, tournament_admin, scorer, viewer
- 요구서 초안: SYSTEM_ADMIN, TOURNAMENT_ADMIN, SCORER, PLAYER, VIEWER

MVP 역할은 SYSTEM_ADMIN, TOURNAMENT_ADMIN, SCORER, PLAYER로 통일한다.
공개 조회가 비로그인 접근이므로 VIEWER는 MVP에서 제외한다.

1.3 일정 기능 우선순위
기존 P1의 일정 관리와 P2의 반자동 일정 배정이 겹친다.
- P1 또는 P2 초기: 코트·시간·심판 수동 배정과 충돌 검사
- 후속 단계: 조건 기반 반자동 추천과 일괄 배정
- 장기 단계: 완전 자동 최적화

1.4 MVP와 다음 스프린트
MVP 업무 흐름에는 코트·시간·심판 배정이 있으나 권장 다음 스프린트에는 없다.
다음처럼 분리한다.
- 1차: 조 구성원 관리 -> 라운드 로빈 -> 수동 경기 생성
- 2차: 점수 임시 저장 -> 결과 확정 -> 순위 자동 계산 -> 공개 반영
- 3차: 코트·시간·심판 수동 배정 -> 충돌 검사

1.5 순위 스냅샷 의미
점수 확정마다 갱신하면서 스냅샷이라고 부르는 것은 의미가 불분명하다.
- MVP의 rankings_snapshots는 선수별 최신 순위 한 건을 유지하는 조회 모델로 사용한다.
- 계산 이력이 필요하면 ranking_calculations 같은 계산 회차 테이블과 순위 이력 테이블을 별도로 추가한다.
- 결과 확정 후 해당 조의 최신 순위를 같은 트랜잭션에서 재계산한다.

1.6 동시 수정 응답
요구사항은 버전 충돌에 409 Conflict를 요구하지만 현재 구현은 IllegalArgumentException을 400으로 반환한다.
VersionConflictException 같은 전용 예외를 만들고 409로 매핑해야 한다.

1.7 점수 규칙 범위
종목별 경기 규칙은 다종목 시스템을 의미하므로 현재 MVP 범위를 넘어선다.
- MVP: 쇼다운 표준 경기 규칙을 고정 적용
- 확장: 대회 또는 단계별 최대 세트 수, 승리 조건과 듀스 규칙 설정

1.8 점수 저장과 결과 확정
현재처럼 점수 저장 즉시 경기를 완료하면 현장 오입력 복구가 어렵다.
다음 상태 흐름으로 분리한다.
- 점수 임시 저장
- 규칙 검증
- 결과 확정
- 승자 및 순위 반영
- 관리자에 의한 결과 재개방과 정정


2. 추가 기능 요구사항

2.1 대회 상태 전이
- 기본 전이는 DRAFT -> PUBLISHED -> RUNNING -> FINISHED -> ARCHIVED로 한다.
- RUNNING 이후 역방향 전환은 관리자만 수행하며 사유를 기록한다.
- 진행 중인 대회의 부문·단계·조 삭제를 제한한다.
- 종료된 대회의 점수 수정은 결과 재개방 절차를 거친다.
- 공개 API에는 정책상 공개 가능한 상태의 대회만 노출한다.

2.2 대진 생성 안전성
- 같은 조에서 동일 선수 조합의 중복 경기를 생성하지 않는다.
- 동일 생성 요청을 재전송해도 경기가 중복되지 않도록 멱등성을 보장한다.
- 생성 결과를 저장 전에 미리 보여준다.
- 생성 중 검증 오류가 발생하면 작업 전체를 롤백한다.
- 기존 확정 경기나 점수가 있으면 재생성을 차단한다.

2.3 경기 규칙
- 최대 세트 수와 승리 필요 세트 수를 정의한다.
- 2대 0 조기 종료와 2대 1 종료를 지원한다.
- 불필요한 추가 세트 입력을 차단한다.
- 세트 유효 점수, 듀스, 기권·실격·부전승 규칙을 정의한다.
- 완료 경기의 선수와 대진은 결과 재개방 전에는 수정할 수 없다.

2.4 시간대
- DB에는 경기 시각을 UTC로 저장한다.
- 입력과 표시는 대회의 IANA 시간대를 사용한다.
- 경기 시간이 대회 기간이나 코트 사용 가능 시간을 벗어나면 차단한다.
- 일광절약시간이 적용되는 지역도 처리한다.

2.5 데이터 무결성
- 같은 선수를 경기 양쪽에 배정할 수 없다.
- 기권·실격 선수는 신규 경기에 배정할 수 없다.
- 대회, 부문, 단계, 조, 선수와 경기의 소속 관계가 일치해야 한다.
- 비활성 코트와 심판은 신규 경기에 배정할 수 없다.
- 참조 데이터가 있는 리소스의 물리 삭제를 차단하고 상태 전환을 안내한다.

2.6 스키마 마이그레이션
- Flyway를 도입해 모든 스키마 변경을 버전으로 관리한다.
- 운영 환경에서는 Hibernate 자동 생성을 사용하지 않고 validate만 허용한다.
- 마이그레이션 실패 시 배포와 애플리케이션 기동을 중단한다.
- 기존 초기 SQL은 Flyway 기준선 마이그레이션으로 전환한다.

2.7 운영 관측성
- 모든 API 요청에 추적 ID를 부여한다.
- 오류 응답과 구조화 로그에 추적 ID를 포함한다.
- 인증 실패, 권한 거부, 점수 저장 실패와 순위 계산 실패를 기록한다.
- 애플리케이션 및 DB 연결 헬스체크를 제공한다.
- 백업 생성뿐 아니라 정기적으로 실제 복구를 검증한다.

2.8 개인정보와 보존
- 공개 가능한 선수 정보 필드를 정의한다.
- 사용자 비활성화와 개인정보 삭제 정책을 정의한다.
- 감사 로그, 경기 데이터와 백업의 보존 기간을 정의한다.
- 백업과 감사 로그 접근 권한을 제한한다.


3. 추가 테스트 시나리오

- 다른 대회의 부문·단계·조·선수 ID를 조합한 요청을 거부한다.
- 동일 선수를 양쪽에 배정한 경기를 거부한다.
- 2대 0 조기 종료와 2대 1 정상 종료를 검증한다.
- 불필요한 추가 세트, 동점 세트와 잘못된 점수 규칙을 거부한다.
- 기권승, 실격과 부전승을 순위에 정확히 반영한다.
- 2인 동률과 3인 이상 동률의 순위 계산과 설명을 검증한다.
- 동일 대진 생성 요청을 재전송해도 경기가 중복되지 않는다.
- 같은 경기 점수를 동시에 저장하면 한 요청만 성공하고 다른 요청은 409를 받는다.
- 완료 경기의 선수 변경과 일반 삭제를 차단한다.
- DRAFT와 ARCHIVED 대회의 공개 API 노출 정책을 검증한다.
- 백업 데이터로 실제 복구가 가능한지 검증한다.
- 키보드만으로 관리자 CRUD와 점수 입력·확정 흐름을 완료한다.


4. 보정된 구현 우선순위

P0. 기반 안정화
- 깨진 소스 인코딩과 빌드 오류 수정
- JPA와 SQL의 기준 모델 확정
- 역할 코드와 인증 방식 통일
- 기본 비밀번호 제거
- 대회 간 소속 검증
- 공통 오류 형식과 409 충돌 응답
- Flyway 기준선 구성

P1. 대회 진행 핵심
- 조 구성원 관리
- 라운드 로빈 대진 생성
- 쇼다운 표준 경기 규칙
- 점수 임시 저장과 결과 확정 분리
- 최신 조별 순위 계산 및 공개 API
- 결과와 순위 변경 감사 로그

P2. 현장 운영
- 코트·시간·심판 수동 배정
- 선수·코트·심판 충돌 검사
- 선수 개인 일정
- 결과 재개방과 정정
- 모바일 기록원 UX

P3. 운영 자동화
- 반자동 일정 추천과 일괄 배정
- CSV 또는 Excel 가져오기·내보내기
- 운영 리포트

P4. 확장
- 실시간 득점 이벤트와 라이브 트래커
- 알림
- 즐겨찾기
- 고급 통계


5. 최종 권고

가장 먼저 완성해야 하는 제품 흐름은 다음과 같다.

대회·부문·선수 준비
-> 조 구성원 배정
-> 라운드 로빈 생성
-> 점수 임시 저장
-> 결과 확정
-> 순위 자동 계산
-> 공개 화면 반영

이 흐름을 완성한 뒤 코트·시간·심판 배정과 일정 자동화를 확장하는 것이 구현 위험과 범위를 가장 안정적으로 통제할 수 있다.
