Showdown 리그전 및 토너먼트 점수 기록 사이트 아키텍처 설계문서

작성 목적:
이 문서는 Showdown 경기의 리그전, 조별 예선, 순위 결정전, 토너먼트, 최종 순위, 경기 점수 기록을 관리하는 웹사이트를 만들기 위한 아키텍처 설계서이다.
문서는 시각장애인이 스크린리더로 읽기 쉽도록 표 형식을 사용하지 않고, 문장과 목록 중심으로 작성한다.

기준 범위:
초기 버전은 대회 운영 MVP를 목표로 한다.
운영 규모는 대회당 선수 20명에서 150명, 코트 5개에서 20개, 운영자 소수 명을 기준으로 한다.
처음부터 협회급 대형 플랫폼으로 만들지는 않지만, 여러 대회를 저장하고 확장할 수 있는 구조로 설계한다.
남자부, 여자부, 청소년부는 각각 별도 부문으로 생성하여 리그전과 토너먼트를 독립적으로 운영할 수 있게 한다.
경기장 또는 테이블 수는 courts 데이터로 지정하고, 활성 경기장 수에 맞춰 자동 일정 생성을 지원한다.
리그전 기본 경기 시간은 30분, 토너먼트와 순위 결정전 기본 경기 시간은 45분으로 설정한다.
대회 등록 시 start_date와 end_date로 특정 날짜 또는 기간을 지정한다.
대회 등록 후 참가 선수와 참가 심판을 등록하고, 자동 스케줄 생성 시 경기마다 심판 1명 또는 2명을 배정할 수 있게 한다.
운영자는 자동 생성된 경기 시간, 경기장, 선수 대진, 심판 배정을 수동으로 변경할 수 있다.

1.1 첫 번째 참고 사이트 분석

참고 주소:
https://showdown-tournament.com/index.php?TN=136&ad=results&HD=1&R=1

사이트 성격:
이 사이트는 Showdown 대회의 공개 결과 조회에 초점을 둔 구조이다.
화면은 매우 단순하고, 메뉴와 텍스트 목록 중심으로 구성되어 있다.
스크린리더로 읽기에 비교적 유리한 구조를 가진다.

주요 화면:
대회 홈 화면.
선수 목록 화면.
그룹 및 결과 화면.
경기 일정 화면.
최종 순위 화면.
관리자 로그인 화면.

관찰한 데이터 구조:
대회에는 이름, 기간, 장소, 언어, 성별 또는 부문 정보가 있다.
선수는 번호, 이름, 시드 번호, 국가 또는 소속 정보를 가진다.
그룹은 GROUP A, GROUP B와 같이 이름을 가진다.
그룹 순위는 순위, 선수 이름, 승패, 승점, 세트 득실, 점수 득실을 표시한다.
경기 일정은 경기 번호, 시간, 테이블 번호, 그룹 또는 라운드명, 선수 1, 선수 2, 경기 승패, 세트 점수, 총 점수, 세트별 결과를 표시한다.
최종 순위는 순위, 선수 이름, 국가 또는 소속을 표시한다.

장점:
공개 화면이 단순하다.
필요한 정보를 빠르게 찾을 수 있다.
HTML이 텍스트 중심이어서 접근성이 좋다.
관리자 기능과 공개 조회 기능의 경계가 명확하다.

아쉬운 점:
개인별 일정이나 다음 경기 확인 기능은 약하다.
라이브 득점 흐름이나 경기 상세 이벤트 기록은 제한적으로 보인다.
화면이 단순한 만큼 모바일에서의 사용자 경험이나 실시간성은 부족할 수 있다.

설계에 반영할 점:
초기 MVP는 이 사이트처럼 선수 목록, 조별 결과, 일정, 최종 순위를 안정적으로 제공해야 한다.
공개 조회 화면은 텍스트 기반으로도 충분히 사용할 수 있어야 한다.
관리자 로그인 이후에만 점수 입력과 대회 설정을 허용해야 한다.

선수를 먼저 선택한 뒤 개인 일정, 다음 경기, 내가 속한 그룹, 전체 일정, 전체 그룹, 최종 순위로 이동할 수 있다.
라이브 티커와 경기 상세 득점 흐름도 제공한다.

주요 화면:
선수 선택 화면.
즐겨찾기 선수 목록.
성별 또는 부문별 선수 목록.

개인 일정 화면.

내 그룹 화면.

전체 그룹 화면.
최종 순위 화면.
라이브 티커 화면.
경기 상세 화면.

관찰한 데이터 구조:
선수는 성별 또는 부문별로 나뉜다.
선수는 국가 코드와 이름을 함께 표시한다.
사용자는 특정 선수를 즐겨찾기로 선택할 수 있다.
일정은 날짜, 시간, 테이블, 경기 ID, 그룹, 선수 1, 선수 2, 결과, 세트별 점수, 심판, 상세 링크를 가진다.
그룹 순위는 순위, 이름, 승점, 경기 수, 세트 득실, 점수 득실, 부분 동률 계산 값을 가진다.
개인 일정은 특정 선수가 참여하는 경기만 따로 보여준다.
다음 경기는 특정 선수의 가장 가까운 예정 경기를 보여준다.
경기 상세는 세트별 득점 이벤트를 순서대로 보여준다.
득점 이벤트에는 서비스 순서, 서버, 득점 또는 실책, 점수 변화가 포함된다.

장점:
선수와 관중이 자기에게 필요한 정보만 빠르게 확인할 수 있다.
개인 일정과 다음 경기 기능은 실제 대회 현장에서 매우 유용하다.
라이브 티커와 득점 이벤트 기록은 현장성과 신뢰성을 높인다.
그룹 순위에 동률 계산용 부분 점수를 따로 보여주어 순위 산정 근거를 설명할 수 있다.

아쉬운 점:
URL에 세션 정보가 포함되어 공유성이 떨어질 수 있다.
화면 일부는 null 같은 문구가 노출되어 데이터 처리 예외가 보인다.
기능이 많아 초기 구현 범위가 커질 수 있다.

설계에 반영할 점:
MVP 이후 확장 기능으로 선수별 일정, 다음 경기, 즐겨찾기, 라이브 티커를 추가한다.
경기 결과뿐 아니라 세트별 점수와 득점 이벤트를 저장할 수 있는 데이터 모델을 준비한다.
동률 처리 계산 결과를 설명 가능한 형태로 저장하거나 재계산할 수 있게 한다.

3. 권장 기술 스택

3.1 1순위 추천

프론트엔드 언어:
TypeScript.

프론트엔드 프레임워크:
Next.js 또는 React.

백엔드 언어:
TypeScript.

백엔드 프레임워크:
NestJS.

데이터베이스:
PostgreSQL.

캐시와 실시간 보조:
Redis.

배포 방식:
초기에는 Docker Compose 기반 단일 서버 배포.
확장 시에는 프론트엔드, 백엔드 API, PostgreSQL, Redis, 정적 파일 서버를 분리.

추천 이유:
TypeScript를 프론트엔드와 백엔드에서 함께 사용할 수 있어 개발 속도와 유지보수성이 좋다.
NestJS는 모듈 구조, 의존성 주입, 인증, 테스트 구성이 안정적이다.
PostgreSQL은 대회, 선수, 경기, 세트, 순위처럼 관계가 복잡한 데이터에 적합하다.
Redis는 실시간 점수 화면, 캐시, 알림, 작업 큐에 활용할 수 있다.


3.2 대안 기술

대안 1:
Java 또는 Kotlin과 Spring Boot.

적합한 경우:
장기 운영을 목표로 하고, 개발팀이 Java 생태계에 익숙한 경우.
대형 협회나 기관 시스템처럼 안정성과 표준화가 중요한 경우.

장점:
트랜잭션 처리와 대규모 백엔드 운영에 강하다.
타입 안정성이 좋다.
운영 경험과 자료가 많다.

단점:
초기 MVP 개발 속도는 TypeScript 스택보다 느릴 수 있다.
프론트엔드와 백엔드 언어가 달라진다.

대안 2:
Python과 FastAPI.

적합한 경우:
빠른 프로토타입 제작이 중요하거나 데이터 처리 자동화가 많은 경우.

장점:
개발 속도가 빠르다.
CSV, Excel, 통계 처리와 잘 맞는다.
API 서버를 가볍게 만들기 좋다.

단점:
대규모 팀 개발에서는 타입 규칙과 프로젝트 구조를 엄격히 잡아야 한다.
복잡한 도메인 로직이 커지면 유지보수 규칙이 필요하다.

대안 3:
PHP와 Laravel.

적합한 경우:
전통적인 서버 렌더링 방식으로 빠르게 공개 조회 사이트를 만들고 싶은 경우.

장점:
관리자 화면과 CRUD 개발이 빠르다.
호스팅 선택지가 많다.
첫 번째 참고 사이트처럼 단순한 구조를 만들기 쉽다.

단점:
실시간 중계와 복잡한 프론트엔드 확장에는 TypeScript 기반 구조보다 불리할 수 있다.


4. 전체 아키텍처

4.1 구성 요소

사용자 브라우저:
관리자, 기록원, 선수, 관중이 접속한다.

프론트엔드 웹 애플리케이션:
공개 조회 화면과 관리자 화면을 제공한다.
접근성을 고려하여 제목, 목록, 버튼, 폼 레이블을 명확히 제공한다.

백엔드 API 서버:
인증, 권한, 대회 관리, 경기 관리, 점수 입력, 순위 계산을 처리한다.

PostgreSQL 데이터베이스:
모든 원본 데이터를 저장한다.
대회, 선수, 경기, 세트, 득점 이벤트, 순위 스냅샷, 감사 로그를 저장한다.

Redis:
캐시와 실시간 알림을 담당한다.
초기 MVP에서는 없어도 동작하게 만들고, 라이브 중계 도입 시 적극 활용한다.

파일 저장소:
CSV 또는 Excel 가져오기 파일, 내보내기 파일, 백업 파일을 저장한다.
초기에는 서버 로컬 볼륨을 사용할 수 있고, 확장 시 S3 호환 스토리지로 이전한다.


4.2 텍스트 흐름도

공개 조회 흐름:
사용자 브라우저
-> 프론트엔드 화면
-> 백엔드 조회 API
-> PostgreSQL
-> 백엔드가 정렬 및 필터링된 결과 반환
-> 프론트엔드가 접근성 좋은 목록 화면으로 표시

경기 결과 입력 흐름:
기록원 브라우저
-> 관리자 로그인
-> 경기 선택
-> 세트별 점수 입력
-> 백엔드 검증
-> match_sets 저장
-> matches 결과 갱신
-> 순위 재계산
-> rankings_snapshots 저장
-> audit_logs 저장
-> 공개 화면에 결과 반영

라이브 득점 확장 흐름:
심판 또는 기록원 브라우저
-> 득점 이벤트 입력
-> 백엔드가 point_events 저장
-> Redis 또는 WebSocket으로 실시간 알림
-> 관중 화면과 라이브 티커 화면 갱신
-> 경기 종료 시 match_sets와 matches 확정


5. 도메인 모델

5.1 대회

테이블명:
tournaments.

역할:
대회 하나의 기본 정보를 저장한다.

주요 필드:
id: 대회 고유 ID.
code: 짧은 대회 코드. 예: slovak-open-2025.
name: 대회 이름.
location: 개최 장소.
start_date: 시작일.
end_date: 종료일.
timezone: 대회 시간대.
status: draft, published, running, finished, archived 중 하나.
default_language: 기본 언어.
created_at: 생성 시각.
updated_at: 수정 시각.

설계 원칙:
모든 하위 데이터는 tournament_id로 대회에 연결한다.
여러 대회를 한 시스템에서 관리할 수 있도록 처음부터 대회 ID를 모든 주요 테이블에 포함한다.
start_date와 end_date로 하루 대회 또는 여러 날 진행되는 대회를 모두 등록할 수 있다.
실제 경기 가능 시간대는 schedule_blocks에서 날짜와 시간 단위로 지정한다.


5.2 부문

테이블명:
divisions.

역할:
남자부, 여자부, 오픈부, 주니어부 같은 경기 부문을 저장한다.

주요 필드:
id: 부문 고유 ID.
tournament_id: 대회 ID.
name: 부문 이름.
code: 부문 코드. 예: M, W, OPEN.
category: 부문 유형. 예: male, female, youth, open.
age_min: 청소년부 또는 연령 제한 부문의 최소 나이.
age_max: 청소년부 또는 연령 제한 부문의 최대 나이.
sort_order: 표시 순서.
is_active: 사용 여부.

설계 원칙:
부문마다 별도의 조, 경기, 순위를 가질 수 있어야 한다.
대회마다 부문 구성이 다를 수 있으므로 전역 고정값으로 만들지 않는다.
남자부는 category를 male로 지정한다.
여자부는 category를 female로 지정한다.
청소년부는 category를 youth로 지정하고 age_min과 age_max를 사용한다.


5.3 선수

테이블명:
players.

역할:
선수의 기본 인물 정보를 저장한다.

주요 필드:
id: 선수 고유 ID.
display_name: 화면 표시 이름.
family_name: 성.
given_name: 이름.
country_code: 국가 코드.
external_ref: 외부 시스템 선수 ID.
created_at: 생성 시각.
updated_at: 수정 시각.

설계 원칙:
선수 기본 정보는 대회와 분리한다.
같은 선수가 여러 대회에 참가할 수 있기 때문이다.


5.4 대회 참가 선수

테이블명:
tournament_players.

역할:
특정 대회와 특정 부문에 참가하는 선수 정보를 저장한다.

주요 필드:
id: 참가 선수 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
player_id: 선수 ID.
seed_no: 시드 번호.
entry_no: 참가 번호.
display_name_override: 대회에서 사용할 표시 이름.
club_name: 소속 클럽.
status: active, withdrawn, disqualified 중 하나.

설계 원칙:
시드 번호와 참가 번호는 대회마다 달라지므로 players가 아니라 tournament_players에 저장한다.


5.5 단계

테이블명:
stages.

역할:
1라운드, 2라운드, 결승 라운드, 순위 결정 라운드 같은 대회 단계를 저장한다.

주요 필드:
id: 단계 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
name: 단계 이름.
stage_type: round_robin, knockout, placement 중 하나.
default_match_duration_minutes: 해당 단계의 기본 경기 시간. 리그전은 30분, 토너먼트와 순위 결정전은 45분을 기본으로 한다.
sort_order: 진행 순서.
status: draft, published, running, finished 중 하나.

설계 원칙:
조별 리그와 토너먼트를 하나의 단계 구조 안에서 함께 표현한다.
각 단계는 여러 그룹 또는 여러 토너먼트 경기를 포함할 수 있다.


5.6 그룹

테이블명:
groups.

역할:
GROUP A, M_A, M_QF1, Final 같은 경기 묶음을 저장한다.

주요 필드:
id: 그룹 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
code: 그룹 코드.
name: 그룹 표시 이름.
group_type: league, knockout, placement 중 하나.
sort_order: 표시 순서.

설계 원칙:
리그 조와 토너먼트 라운드를 모두 그룹으로 표현할 수 있게 한다.
이 방식은 첫 번째 사이트의 GROUP A 구조와 두 번째 사이트의 M_QF1, M_Final 구조를 모두 수용한다.


5.7 그룹 구성원

테이블명:
group_members.

역할:
어떤 선수가 어떤 그룹에 속하는지 저장한다.

주요 필드:
id: 그룹 구성원 고유 ID.
group_id: 그룹 ID.
tournament_player_id: 대회 참가 선수 ID.
slot_no: 그룹 안의 배정 번호.
source_rule: 이전 단계에서 올라온 규칙 설명.

설계 원칙:
선수가 여러 단계의 여러 그룹에 들어갈 수 있으므로 별도 테이블로 관리한다.


5.8 코트 또는 테이블

테이블명:
courts.

역할:
경기가 열리는 테이블 또는 코트를 저장한다.

주요 필드:
id: 코트 고유 ID.
tournament_id: 대회 ID.
name: 표시 이름. 예: Table 1.
sort_order: 표시 순서.
is_active: 사용 여부.

설계 원칙:
경기장 숫자는 courts 테이블의 활성 행 개수로 정한다.
예를 들어 Table 1부터 Table 8까지 8개 행을 만들면 자동 스케줄러는 동시에 최대 8경기를 배정할 수 있다.
특정 경기장이 특정 시간에 사용 불가능하면 schedule_blocks에서 시간대를 제한한다.


5.8.1 경기 시간 규칙

테이블명:
schedule_duration_rules.

역할:
리그전과 토너먼트의 기본 경기 시간을 저장한다.

주요 필드:
id: 경기 시간 규칙 고유 ID.
tournament_id: 대회 ID. null이면 전역 기본 규칙이다.
division_id: 부문 ID. null이면 대회 전체 규칙이다.
stage_type: round_robin, knockout, placement 중 하나.
match_duration_minutes: 경기 시간. 분 단위이다.
break_minutes: 경기 사이 휴식 시간. 분 단위이다.
is_default: 기본 규칙 여부.

설계 원칙:
전역 기본값은 round_robin 30분, knockout 45분, placement 45분이다.
특정 대회나 특정 부문에서 다른 시간이 필요하면 더 구체적인 규칙을 추가한다.
규칙 적용 순서는 부문별 규칙, 대회별 규칙, 전역 기본 규칙 순서이다.


5.8.2 경기 가능 시간대

테이블명:
schedule_blocks.

역할:
자동 일정 생성이 사용할 날짜, 시간, 경기장 사용 가능 범위를 저장한다.

주요 필드:
id: 경기 가능 시간대 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
court_id: 경기장 ID.
starts_at: 시작 시각.
ends_at: 종료 시각.
is_active: 사용 여부.
note: 운영 메모.

설계 원칙:
court_id가 null이면 모든 활성 경기장에 적용되는 공통 시간대이다.
court_id가 지정되면 특정 경기장에만 적용되는 시간대이다.
자동 스케줄러는 schedule_blocks와 courts를 함께 보고 경기장 수와 가능한 시간을 계산한다.


5.8.3 자동 일정 생성 원칙

자동 일정 생성 흐름:
먼저 대회와 부문을 선택한다.
다음으로 리그전 또는 토너먼트 단계를 선택한다.
리그전이면 그룹 안의 선수 조합으로 경기를 만든다.
토너먼트이면 시드 또는 이전 단계 순위에 따라 대진 슬롯을 만든다.
스케줄러는 schedule_duration_rules에서 경기 시간을 찾는다.
스케줄러는 schedule_blocks에서 가능한 시간대와 경기장을 찾는다.
스케줄러는 schedule_official_rules에서 심판 배정 인원을 찾는다.
스케줄러는 officials에서 사용 가능한 심판을 찾는다.
스케줄러는 matches에 scheduled_at, duration_minutes, court_id를 저장한다.
스케줄러는 match_officials에 심판 1명 또는 2명을 저장한다.

충돌 방지 규칙:
같은 선수는 같은 시간대에 두 경기를 배정받을 수 없다.
같은 경기장은 같은 시간대에 두 경기를 배정받을 수 없다.
같은 심판은 같은 시간대에 두 경기에 배정받을 수 없다.
토너먼트 다음 라운드는 이전 라운드가 끝난 뒤에 시작되어야 한다.

수동 변경 원칙:
운영자는 자동 생성된 경기 시간, 경기장, 선수 대진, 심판 배정을 변경할 수 있다.
수동 변경된 경기 일정은 schedule_source를 manual로 저장한다.
수동 변경된 선수 또는 심판 배정은 assignment_source를 manual로 저장한다.
수동 변경 내역은 audit_logs에 변경 전후 데이터와 작업자를 남긴다.


5.9 심판 또는 운영자

테이블명:
officials.

역할:
심판, 기록원, 운영 담당자를 저장한다.

주요 필드:
id: 담당자 고유 ID.
tournament_id: 대회 ID.
name: 이름.
short_code: 짧은 표시 코드.
role_name: referee, scorer, organizer 등 담당 역할.
is_active: 사용 여부.

설계 원칙:
대회 등록 후 참가 심판을 officials에 등록한다.
자동 스케줄러는 role_name이 referee이고 is_active가 true인 담당자를 심판 후보로 사용한다.


5.9.1 심판 배정 규칙

테이블명:
schedule_official_rules.

역할:
자동 일정 생성 시 경기마다 심판을 몇 명 배정할지 저장한다.

주요 필드:
id: 심판 배정 규칙 고유 ID.
tournament_id: 대회 ID. null이면 전역 기본 규칙이다.
division_id: 부문 ID. null이면 대회 전체 규칙이다.
stage_type: round_robin, knockout, placement 중 하나.
required_official_count: 자동 배정 시 반드시 배정할 심판 수. 1명 또는 2명이다.
max_official_count: 운영자가 배정할 수 있는 최대 심판 수. 1명 또는 2명이다.
is_default: 기본 규칙 여부.

설계 원칙:
기본값은 모든 단계에서 심판 1명을 자동 배정하고 최대 2명까지 허용한다.
특정 대회나 특정 부문에서 토너먼트 경기만 심판 2명을 필수로 지정할 수 있다.


5.10 경기

테이블명:
matches.

역할:
경기 하나의 일정, 상태, 결과 요약을 저장한다.

주요 필드:
id: 경기 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
group_id: 그룹 ID.
match_no: 대회 안에서 표시할 경기 번호.
scheduled_at: 경기 예정 시각.
duration_minutes: 실제 배정된 경기 시간. 자동 일정 생성 시 리그전은 기본 30분, 토너먼트와 순위 결정전은 기본 45분이다.
court_id: 코트 ID.
referee_id: 대표 심판 ID.
schedule_source: 일정 배정 출처. auto, manual, imported 중 하나이다.
schedule_locked: 자동 재스케줄링에서 제외할지 여부.
manual_note: 수동 변경 사유 또는 운영 메모.
status: scheduled, running, completed, cancelled, walkover 중 하나.
winner_tournament_player_id: 승자 ID.
player1_match_points: 선수 1 경기 승점 또는 승패 표시 값.
player2_match_points: 선수 2 경기 승점 또는 승패 표시 값.
player1_sets_won: 선수 1 세트 승수.
player2_sets_won: 선수 2 세트 승수.
player1_total_points: 선수 1 총 득점.
player2_total_points: 선수 2 총 득점.
version: 동시 수정 충돌 방지용 버전 번호.
confirmed_at: 결과 확정 시각.

설계 원칙:
matches는 조회 속도를 위해 결과 요약을 가진다.
정확한 원본 점수는 match_sets와 point_events에 저장한다.
결과를 수정하면 version을 확인하여 동시에 두 명이 덮어쓰지 못하게 한다.
schedule_locked가 true인 경기는 자동 스케줄러가 시간과 경기장을 덮어쓰지 않는다.


5.11 경기 참가자

테이블명:
match_players.

역할:
한 경기에 참가하는 선수를 저장한다.

주요 필드:
id: 경기 참가자 고유 ID.
match_id: 경기 ID.
tournament_player_id: 대회 참가 선수 ID.
side: player1 또는 player2.
source_slot: 토너먼트 자동 배정 설명. 예: Winner of M_QF1.
assignment_source: 선수 배정 출처. auto, manual, imported 중 하나이다.

설계 원칙:
일반 경기는 두 명을 저장한다.
향후 복식이나 특수 경기 확장이 필요하면 이 구조가 더 유연하다.
운영자가 선수 대진이나 선수 스케줄을 수동 변경하면 assignment_source를 manual로 저장한다.


5.10.1 경기별 심판 배정

테이블명:
match_officials.

역할:
한 경기에 배정된 심판 1명 또는 2명을 저장한다.

주요 필드:
id: 경기별 심판 배정 고유 ID.
match_id: 경기 ID.
official_id: 심판 ID.
role_name: 경기에서의 역할. 기본값은 referee이다.
position_no: 심판 배정 순서. 1 또는 2이다.
assignment_source: 심판 배정 출처. auto, manual, imported 중 하나이다.
assigned_at: 배정 시각.

설계 원칙:
심판 1명 배정은 position_no 1만 저장한다.
심판 2명 배정은 position_no 1과 position_no 2를 저장한다.
운영자가 심판을 수동 변경하면 assignment_source를 manual로 저장하고 audit_logs에 변경 이력을 남긴다.


5.12 세트 점수

테이블명:
match_sets.

역할:
경기의 세트별 점수를 저장한다.

주요 필드:
id: 세트 고유 ID.
match_id: 경기 ID.
set_no: 세트 번호.
player1_score: 선수 1 점수.
player2_score: 선수 2 점수.
winner_side: player1 또는 player2.
is_tiebreak: 타이브레이크 여부.

설계 원칙:
경기 결과 계산의 가장 중요한 원본 데이터이다.
세트 점수가 바뀌면 matches 결과 요약과 순위를 다시 계산한다.


5.13 득점 이벤트

테이블명:
point_events.

역할:
라이브 득점이나 경기 상세 흐름을 저장한다.

주요 필드:
id: 득점 이벤트 고유 ID.
match_id: 경기 ID.
set_no: 세트 번호.
event_no: 세트 안의 이벤트 순서.
server_side: player1 또는 player2.
service_no: 첫 번째 서비스 또는 두 번째 서비스.
event_type: goal, fault, penalty, correction 중 하나.
scoring_side: 득점한 선수 방향.
player1_score_after: 이벤트 후 선수 1 점수.
player2_score_after: 이벤트 후 선수 2 점수.
created_by_user_id: 입력한 사용자 ID.
created_at: 입력 시각.

설계 원칙:
MVP에서는 필수 입력 대상이 아니지만 테이블은 미리 설계한다.
라이브 티커와 경기 상세 화면은 이 데이터를 사용한다.


5.14 순위 스냅샷

테이블명:
rankings_snapshots.

역할:
특정 시점의 그룹 순위 또는 최종 순위 계산 결과를 저장한다.

주요 필드:
id: 순위 스냅샷 고유 ID.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
group_id: 그룹 ID.
tournament_player_id: 대회 참가 선수 ID.
rank_no: 순위.
matches_played: 경기 수.
wins: 승수.
losses: 패수.
match_points: 승점.
sets_won: 세트 승수.
sets_lost: 세트 패수.
set_difference: 세트 득실.
points_for: 득점.
points_against: 실점.
point_difference: 점수 득실.
tie_break_note: 동률 처리 설명.
calculated_at: 계산 시각.

설계 원칙:
순위는 언제든 재계산 가능해야 한다.
동시에 공개 화면 성능을 위해 마지막 계산 결과를 스냅샷으로 저장한다.


5.15 사용자와 권한

테이블명:
users.

역할:

주요 필드:
id: 사용자 고유 ID.
email: 이메일.
password_hash: 비밀번호 해시.
display_name: 표시 이름.
is_active: 사용 여부.
created_at: 생성 시각.

테이블명:
roles.

역할:
권한 종류를 저장한다.

주요 권한:
system_admin: 시스템 전체 관리자.
tournament_admin: 특정 대회 관리자.
scorer: 점수 입력자.
viewer: 로그인 조회자.

설계 원칙:
대회별 권한을 지원해야 한다.
한 사용자가 어떤 대회에서는 관리자이고 다른 대회에서는 조회자일 수 있다.


5.16 감사 로그

audit_logs.

favorites.

id: 즐겨찾기 고유 ID.

설계 원칙:

6. 순위 계산 규칙

기본 계산 기준:
승리한 선수에게 승점을 부여한다.
패배한 선수는 기본적으로 0점을 받는다.
필요하면 대회 설정에서 승점 규칙을 바꿀 수 있게 한다.

기본 정렬 순서:
1순위는 승점이 높은 선수이다.
2순위는 승수가 많은 선수이다.
3순위는 세트 득실이 높은 선수이다.
4순위는 점수 득실이 높은 선수이다.
5순위는 전체 득점이 높은 선수이다.
6순위는 상호 전적 결과이다.
7순위는 시드 번호 또는 관리자 수동 결정이다.

동률 처리:
동률 선수가 2명인 경우 상호 전적을 우선 적용한다.
동률 선수가 3명 이상인 경우 동률 선수 간 경기만 따로 계산한 부분 승점, 부분 세트 득실, 부분 점수 득실을 적용한다.
그래도 동률이면 대회 규정에 따라 시드 번호 또는 관리자 수동 순위를 적용한다.

설계 원칙:
순위 계산 규칙은 코드에 하드코딩하지 않고 tournament_settings 또는 ranking_rules 형태로 설정 가능하게 한다.
초기 MVP에서는 기본 규칙 하나를 제공하되, 계산 결과에는 tie_break_note를 남겨 사람이 이해할 수 있게 한다.

8. 화면 설계

8.1 공개 화면

대회 홈:
대회 이름, 기간, 장소, 현재 상태, 주요 메뉴를 제공한다.
메뉴는 선수 목록, 일정, 조별 순위, 최종 순위로 구성한다.

선수 목록:
부문별로 선수 목록을 제공한다.
각 선수는 참가 번호, 이름, 국가, 시드 번호를 읽기 쉬운 문장으로 표시한다.

일정 화면:
날짜별로 경기를 묶어서 표시한다.
각 경기는 경기 번호, 시간, 테이블, 그룹, 선수 1, 선수 2, 결과, 세트별 점수를 제공한다.
미완료 경기는 결과 대신 예정 또는 진행 중이라고 표시한다.

조별 순위 화면:
부문과 단계별로 그룹을 표시한다.
각 그룹은 순위, 선수, 승점, 경기 수, 세트 득실, 점수 득실을 문장형 목록으로 표시한다.

최종 순위 화면:
부문별 최종 순위를 표시한다.
각 항목은 순위, 선수 이름, 국가, 소속을 포함한다.

선수별 일정 화면:
특정 선수가 참가하는 경기만 시간순으로 표시한다.
경기 결과와 다음 예정 경기를 쉽게 구분한다.


8.2 관리자 화면

관리자 대시보드:
대회 상태, 등록 선수 수, 생성된 경기 수, 미입력 경기 수, 진행 중 경기 수를 보여준다.

대회 설정 화면:
대회 이름, 기간, 장소, 시간대, 공개 상태를 설정한다.

선수 관리 화면:
선수를 추가, 수정, 삭제한다.
CSV 또는 Excel 가져오기를 지원한다.

조 편성 화면:
부문과 단계별로 그룹을 만들고 선수를 배정한다.

일정 관리 화면:
경기 번호, 시간, 테이블, 심판, 그룹, 선수 배정을 수정한다.

점수 입력 화면:
기록원이 경기를 선택하고 세트별 점수를 입력한다.
입력 즉시 승자, 세트 승수, 총 득점, 순위 영향이 미리 계산되어 표시된다.

결과 확정 화면:
관리자가 경기 결과를 확정한다.
확정 후 수정은 가능하지만 감사 로그가 반드시 남는다.


9. 접근성 설계

기본 원칙:
모든 핵심 기능은 키보드만으로 사용할 수 있어야 한다.
모든 입력 필드는 명확한 레이블을 가져야 한다.
색상만으로 상태를 전달하지 않는다.
화면 제목은 논리적인 순서로 제공한다.
경기 목록과 순위 목록은 스크린리더가 순서대로 읽어도 의미가 통해야 한다.

표 사용 원칙:
시각적 화면에서는 표를 사용할 수 있다.
하지만 모바일과 스크린리더용 대체 화면에서는 문장형 목록을 제공한다.

예시 문장:
경기 12번. 10시 30분. 테이블 2. 그룹 M_A. 선수 1은 김철수. 선수 2는 박영희. 결과는 김철수 승리. 세트 점수는 11대5, 11대8.

상태 안내:
결과 입력 성공, 오류, 저장 중, 순위 재계산 완료 같은 상태는 화면 리더가 읽을 수 있는 라이브 영역으로 제공한다.

URL 공유:
세션 ID가 URL에 노출되지 않도록 한다.
대회 코드와 선수 ID 기반의 공유 가능한 URL을 제공한다.


10. 보안 설계

인증:
관리자와 기록원은 이메일과 비밀번호로 로그인한다.
비밀번호는 강한 해시 알고리즘으로 저장한다.
로그인 세션은 HttpOnly 쿠키를 사용한다.

권한:
공개 조회 API는 로그인 없이 접근 가능하다.
관리자 API는 로그인과 권한 검사를 요구한다.
기록원은 배정된 대회 또는 경기만 입력할 수 있다.
시스템 관리자는 모든 대회를 관리할 수 있다.

데이터 보호:
점수 수정, 경기 삭제, 선수 삭제는 감사 로그를 남긴다.
CSRF 방어를 적용한다.
관리자 API에는 요청 제한을 적용한다.
파일 업로드는 확장자, 크기, 내용 검사를 수행한다.

동시 수정 방지:
matches 테이블에 version 필드를 둔다.
점수 저장 시 클라이언트가 알고 있는 version과 서버 version이 다르면 저장을 거부하고 최신 데이터를 다시 불러오게 한다.


11. 운영 및 배포 설계

초기 배포:
Docker Compose로 프론트엔드, 백엔드, PostgreSQL, Redis를 구성한다.
하나의 VPS 또는 클라우드 서버에서 시작한다.
HTTPS는 Nginx 또는 Caddy로 처리한다.

백업:
PostgreSQL은 매일 자동 백업한다.
대회 진행 중에는 경기 결과 입력 후 짧은 주기로 추가 백업을 수행한다.
백업 파일은 서버 외부 저장소에도 보관한다.

모니터링:
API 오류율을 기록한다.
점수 입력 실패를 기록한다.
DB 연결 상태를 확인한다.
경기 중에는 관리자 대시보드에서 미입력 경기와 충돌 오류를 확인할 수 있게 한다.

로그:
애플리케이션 로그와 감사 로그를 분리한다.
감사 로그는 사용자가 볼 수 있는 운영 기록이다.
애플리케이션 로그는 개발자와 운영자가 장애 분석에 사용한다.


12. 테스트 계획

DB 테스트:
승패 계산을 검증한다.
세트 득실 계산을 검증한다.
점수 득실 계산을 검증한다.
동률 처리 계산을 검증한다.
부전승을 검증한다.
기권을 검증한다.
미완료 경기가 순위에 잘못 반영되지 않는지 검증한다.
점수 재입력 시 순위가 다시 계산되는지 검증한다.

API 테스트:
대회 생성 API를 검증한다.
선수 등록 API를 검증한다.
경기 생성 API를 검증한다.
세트 점수 입력 API를 검증한다.
경기 결과 확정 API를 검증한다.
순위 재계산 API를 검증한다.
일정 조회 API를 검증한다.
선수별 일정 조회 API를 검증한다.

접근성 테스트:
스크린리더가 대회 홈을 순서대로 읽는지 확인한다.
스크린리더가 경기 목록을 한 경기 단위로 이해할 수 있는지 확인한다.
스크린리더가 순위 정보를 선수 단위로 이해할 수 있는지 확인한다.
키보드만으로 점수 입력을 완료할 수 있는지 확인한다.
오류 메시지가 스크린리더로 읽히는지 확인한다.

운영 테스트:
두 명의 기록원이 같은 경기를 동시에 수정할 때 충돌이 감지되는지 확인한다.
경기 결과 수정 시 감사 로그가 남는지 확인한다.
관리자가 순위를 재계산했을 때 공개 화면이 갱신되는지 확인한다.

성능 테스트:
대회당 선수 150명 기준으로 테스트한다.
경기 500개 기준으로 테스트한다.
동시 조회자 500명 기준으로 일정과 순위 조회 응답 시간을 측정한다.


13. 단계별 구현 로드맵

1단계:
프로젝트 기본 구조를 만든다.
PostgreSQL 스키마를 만든다.
사용자 로그인과 권한 구조를 만든다.
대회, 부문, 선수 등록 기능을 만든다.

2단계:
조, 그룹, 그룹 구성원 기능을 만든다.
경기 일정 생성과 수정 기능을 만든다.
공개 선수 목록과 공개 일정 화면을 만든다.

3단계:
세트 점수 입력 기능을 만든다.
경기 결과 자동 계산 기능을 만든다.
조별 순위 계산 기능을 만든다.
공개 조별 순위 화면을 만든다.

4단계:
최종 순위 기능을 만든다.
감사 로그 기능을 만든다.
관리자 대시보드를 만든다.
CSV 또는 Excel 가져오기와 내보내기를 추가한다.

5단계:
선수별 일정과 다음 경기 기능을 추가한다.
즐겨찾기 기능을 추가한다.
라이브 티커와 득점 이벤트 입력 기능을 추가한다.
Server-Sent Events 또는 WebSocket으로 실시간 갱신을 제공한다.


14. MVP 성공 기준

관리자는 대회를 생성할 수 있다.
관리자는 부문과 선수를 등록할 수 있다.
관리자는 조를 만들고 선수를 배정할 수 있다.
관리자는 경기 일정을 만들 수 있다.
기록원은 세트별 점수를 입력할 수 있다.
시스템은 승자와 세트 득실과 점수 득실을 자동 계산한다.
시스템은 조별 순위를 자동 계산한다.
공개 사용자는 선수 목록, 일정, 조별 순위, 최종 순위를 조회할 수 있다.
스크린리더 사용자는 주요 정보를 문장 순서대로 이해할 수 있다.
점수 수정과 결과 확정은 감사 로그에 남는다.


15. 최종 권장안

가장 추천하는 방향:
TypeScript, Next.js, NestJS, PostgreSQL 조합으로 시작한다.
초기에는 Docker Compose 단일 서버 배포로 운영한다.
MVP에서는 경기 결과 입력과 순위 자동 계산을 가장 중요하게 만든다.
라이브 티커와 득점 이벤트 기록은 데이터 구조를 미리 준비하되 2차 개발로 분리한다.

DBA 관점의 핵심 권장 사항:
경기 결과의 원본은 match_sets와 point_events에 둔다.
matches에는 조회 성능을 위한 결과 요약만 둔다.
순위는 재계산 가능해야 하며, 공개 화면 성능을 위해 rankings_snapshots에 저장한다.
모든 주요 테이블에는 tournament_id를 포함하여 여러 대회를 안정적으로 지원한다.
점수 수정과 순위 재계산은 반드시 audit_logs에 남긴다.
동시 입력 충돌을 막기 위해 version 필드를 사용한다.
접근성 좋은 공개 조회 화면은 처음부터 핵심 요구사항으로 취급한다.

이 설계의 핵심 결론:
첫 번째 참고 사이트의 단순하고 안정적인 대회 조회 구조를 MVP의 기본으로 삼는다.
두 번째 참고 사이트의 선수별 일정, 다음 경기, 즐겨찾기, 라이브 티커, 득점 이벤트 기록은 확장 기능으로 반영한다.
데이터베이스는 경기 결과와 순위 계산의 신뢰성을 보장하는 중심 역할을 해야 한다.
