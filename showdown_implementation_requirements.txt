Showdown 구현 현황 분석 및 구현요구서 v0.1

작성 목적
현재 프로젝트의 실제 구현 범위를 분석하고, 향후 브레인스토밍과 개발 우선순위 결정에 사용할 구현요구서를 정의한다.


1. 현재 구현 상태

1.1 전체 수준
현재 프로젝트는 기본 CRUD와 점수 입력이 연결된 프로토타입 단계이다.
대회 운영의 핵심 흐름인 조 편성, 자동 대진·일정 생성, 순위 계산, 실제 사용자 인증은 아직 구현되지 않았다.

1.2 영역별 상태
- 공개 대회·선수·조·경기 조회: 부분 완료. API와 화면이 연결되어 있다.
- 관리자 대회·부문·선수 CRUD: 완료. 실제 REST API를 사용한다.
- 단계·조·심판·경기 CRUD: 완료. 다만 관계 유효성 검증이 부족하다.
- 세트 점수 입력: 부분 완료. 결과·승자 계산과 낙관적 잠금을 지원한다.
- 조별·최종 순위: 미구현. 화면 골격만 있고 API와 계산 로직은 없다.
- 조 선수 배정: 미구현. group_members는 SQL에만 존재한다.
- 코트·경기장 관리: 미구현. 관련 테이블은 SQL에만 존재한다.
- 심판 경기 배정: 미구현. 실제 관계 기반 배정 기능이 없다.
- 자동 대진 생성: 미구현. 라운드 로빈과 토너먼트 생성기가 없다.
- 자동 일정 생성: 미구현. 시간·코트·선수·심판 충돌 계산이 없다.
- 실시간 득점: 미구현. point_events는 SQL에만 존재한다.
- 감사 로그: 미구현. audit_logs는 SQL에만 존재한다.
- 사용자 및 RBAC: 부분 완료. DB 모델은 있으나 실제 인증은 인메모리 Basic Auth이다.
- 즐겨찾기·개인 일정: 미구현. SQL 설계만 존재한다.
- 테스트: 부분 완료. 대표 통합 흐름 한 개와 제한적인 E2E 시나리오가 존재한다.


2. 분석 중 발견된 핵심 문제

2.1 기술 스택 불일치
- 프로젝트 지침에는 PostgreSQL과 Prisma ORM이 명시되어 있다.
- 실제 백엔드는 Spring Boot와 Spring Data JPA를 사용한다.
- 백엔드 테스트도 Jest와 Supertest가 아닌 JUnit과 MockMvc를 사용한다.
- Spring Boot를 유지한다면 Prisma 및 Jest 요구사항을 실제 스택에 맞게 수정해야 한다.

2.2 DB 설계와 애플리케이션 구현의 격차
SQL에는 코트, 조 구성원, 일정 규칙, 심판 배정, 순위, 득점 이벤트, 감사 로그 등이 정의되어 있다.
그러나 다수 테이블에 대응하는 JPA Entity, Repository, Service, API와 화면이 없다.

2.3 순위 기능 미구현
- 프런트엔드의 rankings 데이터는 항상 빈 배열이다.
- 공개 순위 조회 API와 순위 계산 서비스가 없다.
- 순위 페이지 일부에는 깨진 한글과 JSX 문법 손상이 있다.

2.4 운영용으로 부적합한 인증
- 개발용 기본 계정과 비밀번호가 설정에 존재한다.
- 로그인 화면은 서버 인증 없이 입력값을 sessionStorage에 저장한다.
- DB 사용자·역할과 Spring Security 인증이 연결되지 않았다.
- CSRF가 비활성화되어 있다.
- 선수 로그인 후 이동 대상으로 사용하는 /players 경로가 구현되어 있지 않다.

2.5 대회 간 데이터 혼입 가능성
선수, 단계, 조, 경기 생성 시 참조 데이터가 동일한 대회에 속하는지 검증하지 않는다.
다른 대회의 UUID를 전달하면 잘못된 교차 참조가 생성될 수 있다.

2.6 삭제 및 복구 정책 부족
설계 문서는 상태 변경 중심의 논리 삭제를 권장하지만 현재 CRUD는 물리 삭제를 사용한다.
경기에서 참조하는 선수·부문·단계 등을 삭제할 때 운영 복구 정책과 사용자 안내가 부족하다.


3. 제품 목표

관리자가 쇼다운 대회를 생성하고 참가자·부문·조·경기 일정을 구성하며, 기록원이 경기 결과를 입력하면 조별 순위와 최종 순위가 자동 계산되어 공개 화면에 반영되는 시스템을 구축한다.


4. 핵심 사용자

- 시스템 관리자: 사용자와 전체 대회를 관리한다.
- 대회 관리자: 대회 구성, 선수, 조, 일정과 결과를 관리한다.
- 심판·기록원: 담당 경기를 조회하고 점수를 입력한다.
- 선수: 개인 일정과 경기 결과를 조회한다.
- 관중: 공개 일정, 결과와 순위를 조회한다.


5. 기능 요구사항

FR-01 인증 및 권한
- DB 기반 로그인을 제공한다.
- 비밀번호는 BCrypt 등 안전한 단방향 해시로 저장한다.
- 역할은 SYSTEM_ADMIN, TOURNAMENT_ADMIN, SCORER, PLAYER, VIEWER로 구분한다.
- 대회 관리 권한은 대회별로 제한한다.
- 로그인 성공 후에만 세션 또는 토큰을 저장한다.
- 기본 계정과 기본 비밀번호는 운영 환경에서 허용하지 않는다.
- 다른 대회의 관리자가 해당 대회 데이터를 수정하면 403을 반환한다.

FR-02 대회 관리
- 대회 생성, 수정, 게시, 진행, 종료, 보관 상태를 지원한다.
- 관리자가 여러 대회 중 작업 대회를 선택할 수 있어야 한다.
- 대회 코드 중복을 방지한다.
- 대회 기간과 시간대를 검증한다.
- 공개 화면에는 게시 가능한 상태의 대회만 노출한다.

FR-03 참가자 및 부문 관리
- 선수 기본 정보와 대회 참가 정보를 분리한다.
- 참가 번호, 시드, 소속, 참가 상태를 관리한다.
- 동일 대회와 부문 안에서 참가 번호 중복을 방지한다.
- CSV 또는 Excel 일괄 등록 기능을 후속 범위로 제공한다.
- 참조 중인 참가자는 물리 삭제보다 기권·실격 상태 전환을 우선한다.

FR-04 단계 및 조 편성
- 리그, 토너먼트, 순위 결정 단계를 지원한다.
- 관리자가 선수를 조에 직접 배정할 수 있어야 한다.
- 시드 기반 자동 조 편성을 지원한다.
- 한 조 안에서 슬롯과 선수가 중복되지 않아야 한다.
- 부문, 단계, 조와 선수는 반드시 동일 대회 소속이어야 한다.

FR-05 대진 생성
- 리그는 조 구성원을 기준으로 라운드 로빈 경기를 생성한다.
- 토너먼트는 시드 또는 이전 단계 순위를 기반으로 대진을 생성한다.
- 재생성 시 기존 점수 데이터가 있으면 실행을 차단하거나 명시적 확인을 요구한다.
- 부전승과 기권승을 지원한다.
- 생성 결과를 저장하기 전에 관리자가 미리 볼 수 있어야 한다.

FR-06 코트 및 일정
- 대회별 코트 목록과 사용 가능 시간을 관리한다.
- 경기 유형별 기본 경기 시간과 휴식 시간을 설정한다.
- 동일 시간의 선수, 코트와 심판 중복 배정을 방지한다.
- 자동 배정 결과를 관리자가 수동 수정하고 잠글 수 있어야 한다.
- 수동 변경 사유를 기록한다.
- 1차 MVP는 완전 자동 최적화보다 충돌 검사 기반 반자동 배정을 우선한다.

FR-07 심판 배정
- 경기당 1명 또는 2명의 심판·기록원을 배정한다.
- 동일 심판의 같은 시간대 중복 배정을 방지한다.
- 기록원은 자신에게 배정된 경기만 수정할 수 있어야 한다.
- 관리자는 자동 배정 결과를 변경할 수 있어야 한다.

FR-08 점수 및 경기 상태
- 경기 상태를 예정, 진행, 완료, 취소, 기권승으로 관리한다.
- 세트 점수의 음수, 동점과 중복 세트 번호를 차단한다.
- 종목별 승리 세트 수와 점수 규칙을 설정할 수 있어야 한다.
- 동시 수정 시 최신 데이터를 다시 조회하도록 409 Conflict를 반환한다.
- 확정된 결과를 수정할 때 사유와 감사 로그를 남긴다.
- 점수 저장, 경기 결과 확정과 순위 재계산의 트랜잭션 경계를 명확히 한다.

FR-09 순위 계산
- 기본 정렬 순서는 승점, 승수, 세트 득실, 점수 득실, 총 득점 순으로 한다.
- 2인 동률은 상대 전적을 적용한다.
- 3인 이상 동률은 동률 선수 사이의 경기만 재계산한다.
- 계산 근거를 tieBreakNote에 저장하여 공개한다.
- 결과 저장 후 해당 조 순위를 재계산한다.
- 단계 종료 후 최종 순위 스냅샷을 생성한다.
- 순위 재계산은 중복 실행해도 같은 결과를 생성해야 한다.

FR-10 공개 조회
- 공개 가능한 상태의 대회만 비로그인 사용자에게 노출한다.
- 선수, 조, 일정, 결과, 조별 순위와 최종 순위를 제공한다.
- 빈 목록과 API 오류를 명확하게 구분한다.
- 날짜, 부문, 경기 상태와 선수 기준 필터를 제공한다.
- URL로 특정 대회 화면을 직접 공유할 수 있어야 한다.

FR-11 선수 포털
- 로그인한 선수와 연결된 경기만 모아 조회한다.
- 다음 경기, 코트, 예정 시간과 상대 선수를 표시한다.
- 완료 경기 결과와 개인 대회 성적을 제공한다.
- 즐겨찾기는 비로그인 사용자도 익명 키로 사용할 수 있도록 확장할 수 있다.

FR-12 감사 및 운영
- 선수 변경, 일정 변경, 점수 수정, 결과 확정과 순위 재계산을 기록한다.
- 변경 전후 값을 JSON으로 보관한다.
- 관리자가 감사 로그를 조건별로 검색할 수 있어야 한다.
- PostgreSQL 백업과 복구 절차를 문서화한다.
- 주요 운영 변경은 사용자, 시간, 대상과 사유를 추적할 수 있어야 한다.


6. 비기능 요구사항

NFR-01 접근성
- 시맨틱 HTML을 사용한다.
- 모든 대화형 요소는 키보드로 사용할 수 있어야 한다.
- focus-visible 상태를 명확하게 표시한다.
- WCAG AA 기준 4.5:1 이상의 색상 대비를 유지한다.
- 비동기 저장 결과와 오류는 스크린리더 알림 영역에 전달한다.

NFR-02 보안
- 비밀번호, DB 접속 정보와 토큰 키는 환경변수로만 관리한다.
- 운영 CORS 출처는 환경변수로 제한한다.
- 모든 대회 종속 API에서 대회 소유 관계를 검증한다.
- 클라이언트에는 내부 예외와 SQL 정보를 노출하지 않는다.
- 인증 방식에 맞는 CSRF 또는 토큰 보호 정책을 적용한다.

NFR-03 API 및 오류 처리
- API 오류는 상태 코드, 오류 코드, 사용자 메시지와 추적 ID를 포함하는 공통 형식으로 반환한다.
- 유효성 오류는 필드별로 반환한다.
- 리소스 없음은 404, 권한 부족은 403, 동시 수정은 409를 사용한다.

NFR-04 성능
- 공개 목록 API는 페이지네이션을 지원한다.
- 경기 일정과 순위 조회에 필요한 인덱스를 유지한다.
- 반복적인 N+1 조회를 방지한다.
- 대회당 선수 150명, 코트 20개 수준에서 운영 가능한 응답 시간을 확보한다.

NFR-05 테스트 및 품질
- 핵심 서비스 로직에 단위 테스트를 작성한다.
- 정상, 실패, 권한, 데이터 소속과 동시성 시나리오를 통합 테스트한다.
- 공개 조회, 관리자 CRUD, 점수 입력과 순위 반영을 E2E로 검증한다.
- 프런트 빌드, 타입 검사, 린트와 접근성 검사를 CI에서 실행한다.
- 백엔드 빌드와 테스트를 CI에서 실행한다.


7. 우선순위별 구현 계획

P0. 현재 골격 안정화
- 깨진 순위 페이지와 소스 인코딩을 수정한다.
- 인증 스택을 결정하고 기본 비밀번호를 제거한다.
- 대회 간 참조 데이터 검증을 추가한다.
- 공통 API 오류와 409 충돌 응답을 정리한다.
- JPA 모델과 SQL 스키마 중 무엇을 기준 모델로 삼을지 확정한다.
- Spring Boot를 유지할 경우 프로젝트 테스트·ORM 규칙을 실제 스택과 일치시킨다.

P1. MVP 완성
- 조 구성원 관리 기능을 구현한다.
- 라운드 로빈 대진 생성을 구현한다.
- 코트, 심판과 일정 관리를 구현한다.
- 순위 계산 서비스와 공개 순위 API를 구현한다.
- 감사 로그를 구현한다.
- 관련 단위·통합·E2E 테스트를 확대한다.

P2. 현장 운영성 강화
- 선수 개인 일정을 구현한다.
- 충돌 검사 기반 반자동 일정 배정을 구현한다.
- CSV 또는 Excel 가져오기를 구현한다.
- 경기 결과 수정 승인과 변경 사유 입력을 구현한다.
- 모바일 기록원 UX를 개선한다.

P3. 확장 기능
- 실시간 득점 이벤트를 구현한다.
- 라이브 트래커를 구현한다.
- 경기 알림을 구현한다.
- 즐겨찾기를 구현한다.
- 통계와 운영 리포트를 구현한다.


8. 권장 MVP 업무 흐름

대회 생성
-> 부문 생성
-> 참가 선수 등록
-> 단계 생성
-> 조 생성 및 선수 배정
-> 라운드 로빈 대진 생성
-> 코트·시간·심판 배정
-> 기록원 점수 입력
-> 경기 결과 확정
-> 조별 순위 자동 계산
-> 공개 화면 반영

이 흐름이 처음부터 끝까지 실제 데이터로 동작하는 것을 1차 MVP 완료 기준으로 삼는다.


9. 브레인스토밍 핵심 논점

9.1 MVP의 중심
대회 설정 CRUD를 넓히는 것보다 조 편성, 대진 생성, 점수 입력, 순위 계산의 한 흐름을 먼저 완성하는 것을 권장한다.

9.2 인증 방식
- 소규모 웹 운영과 서버 중심 구성이면 세션 기반 인증이 단순하다.
- 외부 앱이나 다중 클라이언트 연동까지 고려하면 JWT 기반 인증이 적합하다.
- 현재 Basic Auth는 개발 확인용으로만 사용한다.

9.3 일정 자동화 수준
초기에는 복잡한 완전 자동 최적화보다 관리자가 배정하고 시스템이 충돌을 탐지하는 반자동 방식을 권장한다.

9.4 데이터 모델 기준
SQL 설계가 구현보다 앞서 있으므로 모든 설계 테이블을 즉시 구현할지, MVP에 필요한 테이블만 단계적으로 활성화할지 결정해야 한다.

9.5 권장 다음 스프린트
조 구성원 관리 -> 라운드 로빈 생성 -> 순위 자동 계산 순으로 구현한다.
이 세 기능이 연결되면 현재 시스템은 CRUD 프로토타입에서 실제 대회 운영 제품으로 전환된다.
