1. TDD 기본 원칙

TDD-01 Red-Green-Refactor
- 모든 신규 비즈니스 기능은 실패하는 테스트 작성(Red), 테스트를 통과하는 최소 구현(Green), 중복과 구조 개선(Refactor) 순서로 개발한다.
- 테스트가 요구사항에 맞게 실패하는 것을 확인하기 전에 제품 코드를 구현하지 않는다.
- Green 단계에서는 현재 실패 테스트를 통과하는 데 필요한 최소 코드만 작성한다.
- Refactor 단계에서는 외부 동작을 변경하지 않으며 전체 테스트가 계속 통과해야 한다.

TDD-02 버그 수정
- 모든 버그는 문제를 재현하는 실패 테스트를 먼저 작성한다.
- 수정 후 재현 테스트와 기존 회귀 테스트가 모두 통과해야 한다.
- 재현 테스트 없이 증상만 수정하거나 실패 테스트를 삭제해 해결한 것으로 처리하지 않는다.

TDD-03 테스트 설계
- 테스트는 구현 내부가 아니라 사용자 관점에서 관찰 가능한 입력, 출력, 상태 변화와 오류를 검증한다.
- 한 테스트는 하나의 명확한 동작 또는 규칙을 검증한다.
- Given-When-Then 또는 Arrange-Act-Assert 구조를 일관되게 사용한다.
- 테스트 이름에는 조건, 실행과 기대 결과가 드러나야 한다.
- 테스트는 실행 순서와 다른 테스트의 데이터에 의존하지 않아야 한다.
- 시간, UUID, 외부 API, 파일 시스템과 현재 사용자 같은 비결정적 의존성은 주입 가능한 구조로 설계한다.


2. 계층별 TDD 요구사항

TDD-04 Domain 및 Service 단위 테스트
- 순위 계산, 동률 처리, 라운드 로빈 생성, 경기 규칙과 상태 전이를 단위 테스트로 우선 개발한다.
- 대회 간 소속 검증, 동일 선수 중복 배정, 기권·실격 선수 배정 금지와 삭제 제한을 검증한다.
- 정상 경로뿐 아니라 경계값, 잘못된 상태 전이와 실패 결과를 검증한다.
- 핵심 도메인 객체를 과도하게 모킹하지 않고 실제 객체와 값 객체를 사용한다.

TDD-05 Repository 테스트
- Repository와 JPA 매핑은 실제 PostgreSQL과 동일한 환경을 사용하는 통합 테스트로 검증한다.
- 외래 키, unique, check constraint, enum, 낙관적 잠금과 주요 조회 쿼리를 검증한다.
- H2 전용 동작에 의존하지 않으며 PostgreSQL 차이를 검증할 수 있도록 Testcontainers 도입을 권장한다.
- 테스트 데이터는 각 테스트가 생성하고 종료 후 격리 또는 롤백한다.

TDD-06 Controller 및 보안 통합 테스트
- 요청 DTO 검증, 인증, 역할별 인가와 대회별 권한 범위를 검증한다.
- 성공 상태 코드와 함께 400, 401, 403, 404, 409 응답을 검증한다.
- 공통 오류 응답의 오류 코드, 사용자 메시지, 필드 오류와 추적 ID를 검증한다.
- 다른 대회의 리소스 ID를 조합한 요청이 거부되는지 검증한다.

TDD-07 프런트엔드 테스트
- 사용자 입력과 제출, API 성공·실패, 로딩, 빈 상태와 동시 수정 충돌 화면을 컴포넌트 테스트로 검증한다.
- 버튼 이름, 레이블과 역할을 기준으로 요소를 선택하여 접근 가능한 UI를 유도한다.
- 키보드 탐색, focus-visible, 오류 연결과 aria-live 안내를 검증한다.
- API 호출 자체보다 사용자에게 표시되는 결과와 상호작용을 우선 검증한다.

TDD-08 E2E 테스트
- E2E는 대회 생성부터 공개 순위 반영까지의 핵심 사용자 흐름을 검증한다.
- 모든 세부 규칙을 E2E에 중복하지 않고 Domain 및 Service 테스트에 둔다.
- 최소 핵심 흐름은 관리자 로그인, 대회·선수·조 준비, 대진 생성, 점수 임시 저장, 결과 확정, 순위 공개이다.
- E2E 테스트 데이터는 고유 식별자를 사용하고 반복 실행할 수 있어야 한다.


3. 요구사항별 필수 테스트 목록

3.1 IBSA 개인전 점수 규칙 기준
- 근거 문서는 IBSA Showdown Rules 2022-2025 영문 원문이며, 규칙 5, 7, 8, 15와 19를 기준으로 한다.
- MVP는 개인전만 지원하고 경기 형식은 1세트제, 3세트제 또는 5세트제 중 하나로 설정한다.
- 경기 승리에 필요한 세트 수는 각각 1세트, 2세트 또는 3세트이며, 필요한 세트를 먼저 획득한 선수가 경기에서 승리한다.
- 일반 세트는 한 선수가 최소 11점에 도달하고 상대보다 2점 이상 앞설 때 종료한다. 10대 10 이후에도 2점 차가 날 때까지 계속하며 최고 점수 상한은 두지 않는다.
- 정규 IBSA 승인 대회에는 시간제 경기를 적용하지 않는다. 비승인 대회의 선택형 시간제 경기는 MVP에서 제외하고 후속 기능으로 분리한다.
- 경기 포기(GIVING_UP)는 부상 등으로 선수가 자발적으로 경기를 중단한 경우다. 포기 선수의 이미 확정된 세트와 모든 기득점은 유지하고, 상대 선수에게 경기를 끝내는 데 필요한 최소 결과를 부여한다.
- 포기 시 진행 중 세트는 포기 선수의 현재 점수를 유지하고 상대 점수를 최소 11점이면서 포기 선수보다 2점 높은 점수까지 올려 상대의 세트 승리로 확정한다. 상대에게 추가 승리 세트가 필요하면 나머지는 11대 0으로 부여한다.
- 몰수패(DEFAULT_LOSS)는 규칙 위반, 중대한 부정행위 또는 5분을 초과한 지각 등으로 패배가 선언된 경우다. 패자의 기존 세트와 득점을 모두 무효화하고 상대에게 경기 형식상 필요한 모든 세트를 11대 0으로 부여한다.
- 경기 포기, 몰수패와 대진표상의 부전승(BYE)은 원인과 통계 의미가 다르므로 서로 다른 종료 사유로 저장한다. BYE는 실제 경기가 아니므로 세트와 득점을 생성하지 않는다.
- 포인트 이벤트를 구현할 때 골은 2점, 일반 플레이 실수는 상대에게 1점, 경고 이후 또는 즉시 부과되는 페널티는 상대에게 2점으로 기록한다. 포인트 이벤트 입력은 MVP 후속 범위이며 MVP에서는 확정 세트 점수의 규칙 검증을 우선한다.
- 팀전의 31점 및 2점 차 규칙, 선수 교대와 3회 연속 서브 규칙은 MVP 범위에서 제외한다.

TDD-09 조 구성원과 대진
- 같은 조 안에서 선수와 슬롯 중복을 거부한다.
- 다른 대회 또는 부문의 선수를 조에 배정하면 거부한다.
- N명의 라운드 로빈 경기 수가 N x (N - 1) / 2인지 검증한다.
- 홀수 인원에서 라운드별 휴식 선수가 올바르게 배정되는지 검증한다.
- 동일 대진 생성 요청을 재실행해도 경기가 중복되지 않는지 검증한다.

TDD-10 점수와 결과 확정
- 1세트제의 1대 0, 3세트제의 2대 0과 2대 1, 5세트제의 3대 0부터 3대 2까지 경기 결과를 검증한다.
- 11대 9, 12대 10과 15대 13은 유효하고 11대 10, 10대 9, 동점과 음수 점수는 무효인지 검증한다.
- 최고 점수 상한 없이 2점 차가 날 때까지 세트가 계속되는지 검증한다.
- 중복 세트 번호, 연속되지 않은 세트 번호, 승리 필요 세트 수를 채운 뒤의 불필요한 추가 세트를 거부한다.
- 경기 형식이 1, 3 또는 5세트제가 아니면 거부한다.
- 임시 저장 단계에서는 승자와 순위가 확정되지 않는지 검증한다.
- 결과 확정 단계에서만 승자와 순위가 반영되는지 검증한다.
- 경기 포기 시 확정 세트와 기득점을 유지하면서 상대에게 최소 결과만 부여하는지 검증한다.
- 몰수패 시 패자의 기존 세트와 득점을 무효화하고 경기 형식에 맞는 11대 0 세트를 부여하는지 검증한다.
- BYE에는 세트 점수와 경기 통계를 생성하지 않는지 검증한다.
- 포기, 몰수패와 BYE 종료 사유가 결과 응답과 감사 로그에 구분되어 기록되는지 검증한다.
- 동일 버전의 동시 수정 중 하나만 성공하고 다른 요청은 409를 반환하는지 검증한다.

TDD-11 순위 계산
- 승점, 승수, 세트 득실, 점수 득실과 총 득점 우선순위를 검증한다.
- 2인 동률에서 상대 전적을 적용하는지 검증한다.
- 3인 이상 동률에서 동률 선수 사이 경기만 재계산하는지 검증한다.
- 기권승, 실격과 부전승의 순위 반영을 검증한다.
- 같은 경기 결과로 순위를 여러 번 계산해도 동일한 최신 순위가 유지되는지 검증한다.
- tieBreakNote가 실제 계산 근거와 일치하는지 검증한다.

TDD-12 일정과 배정
- 동일 시간에 한 선수가 두 경기에 배정되지 않는지 검증한다.
- 같은 코트와 심판의 시간 중복을 거부하는지 검증한다.
- 비활성 코트와 심판을 배정할 수 없는지 검증한다.
- 경기 시간이 대회 기간과 사용 가능 시간을 벗어나면 거부하는지 검증한다.
- UTC 저장값이 대회 시간대로 정확히 표시되는지 검증한다.


4. TDD 작업 단위와 산출물

TDD-13 작업 시작 조건
- 작업은 하나 이상의 요구사항 ID와 인수 조건을 가져야 한다.
- 구현 전 정상 경로, 경계값과 실패 경로의 테스트 목록을 작성한다.
- 데이터 모델이나 공개 API가 변경되면 호환성과 마이그레이션 테스트를 먼저 정의한다.

TDD-14 커밋과 Pull Request
- 기능 구현 커밋에는 해당 요구사항을 검증하는 테스트를 함께 포함한다.
- 가능하면 Red, Green, Refactor 흐름을 검토할 수 있는 작은 커밋으로 분리한다.
- Pull Request에는 요구사항 ID, 테스트 시나리오, 최초 실패 확인, 구현 결과와 실행한 검증 명령을 기록한다.
- 코드 리뷰에서는 테스트가 요구사항을 충분히 설명하는지와 구현 세부사항에 과결합되지 않았는지 확인한다.

TDD-15 완료 정의
- 요구사항에 연결된 단위, 통합 또는 E2E 테스트가 존재한다.
- 신규 테스트가 구현 전 의도한 이유로 실패했음이 확인된다.
- 신규 테스트와 전체 회귀 테스트가 통과한다.
- 프런트엔드 타입 검사와 린트, 백엔드 빌드가 통과한다.
- 중복 제거와 명명 개선을 포함한 리팩터링이 완료된다.
- 공개 API 또는 스키마 변경 시 문서와 마이그레이션이 함께 갱신된다.


5. CI 품질 게이트

TDD-16 병합 조건
- 백엔드 단위·통합 테스트가 모두 통과해야 한다.
- 프런트엔드 단위·컴포넌트 테스트, 타입 검사와 린트가 통과해야 한다.
- 핵심 E2E 테스트가 통과해야 한다.
- 실패 테스트를 비활성화하거나 삭제하여 CI를 통과시키지 않는다.
- 불안정한 테스트는 원인을 해결하며 단순 재실행으로 성공 처리하지 않는다.

TDD-17 커버리지
- 커버리지는 테스트 품질을 보조하는 지표로 사용한다.
- 핵심 Domain 및 Service 변경 라인의 커버리지는 90% 이상을 목표로 한다.
- 전체 수치 달성을 위한 무의미한 테스트보다 요구사항과 위험 시나리오 충족을 우선한다.
- 커버리지 예외가 필요하면 Pull Request에 대상과 사유를 기록한다.


6. 프로젝트 테스트 도구 기준

- Spring Boot 백엔드는 JUnit 5, Spring Boot Test, MockMvc와 Testcontainers PostgreSQL을 사용한다.
- 프런트엔드는 React Testing Library 계열의 컴포넌트 테스트 도구와 Playwright E2E를 사용한다.
- 현재 프로젝트 지침의 Jest 및 Supertest 요구는 Spring Boot 백엔드와 맞지 않으므로 프런트엔드 테스트 또는 Node.js 서비스에만 적용한다.
- 테스트 명령은 로컬과 CI에서 동일하게 실행될 수 있어야 한다.


7. 적용 우선순위

P0
- 테스트 도구와 디렉터리 규칙 확정
- CI에 백엔드 테스트, 프런트 테스트, 타입 검사와 린트 추가
- 공통 테스트 데이터 빌더와 PostgreSQL 통합 테스트 기반 구성

- 컴포넌트 테스트는 작성하지 못했다. 이 프로젝트에는 React Testing Library/Jest/Vitest 등 프런트엔드 단위·컴포넌트 테스트 도구 자체가 아직 없다(package.json에 @playwright/test만 존재). 대신 typecheck/lint 통과와 dev 서버 구동 후 로그인 전 상태 화면이 정상 렌더링되고 콘솔 오류가 없음을 수동으로 확인했다. 실제 백엔드(PostgreSQL)가 이 세션 환경에 없어 409 흐름 자체의 e2e 재현은 하지 못했다.
- 후속 작업: 프런트엔드 컴포넌트 테스트 도구 도입(TDD-07 선행 조건) 후 409 흐름의 회귀 테스트를 추가해야 한다.
