Showdown PostgreSQL 데이터베이스 설계문서

작성 목적:
이 문서는 Showdown 리그전 및 토너먼트 기록 사이트의 PostgreSQL 데이터베이스 구조를 설명한다.
문서는 시각장애인이 스크린리더로 읽기 쉽도록 표를 사용하지 않고 문장과 짧은 목록으로 작성한다.
테이블명과 컬럼명은 애플리케이션 개발과 SQL 작성에 맞게 영어 snake_case를 사용한다.

기준 DBMS:
PostgreSQL 15 이상.

기준 아키텍처:
프론트엔드는 Next.js 또는 React를 사용한다.
백엔드는 NestJS와 TypeScript를 사용한다.
데이터베이스는 PostgreSQL을 사용한다.
캐시와 실시간 알림은 Redis를 추가할 수 있다.


1. 전체 데이터 모델 요약

이 데이터베이스는 대회, 부문, 선수, 조, 경기, 세트 점수, 득점 이벤트, 순위, 사용자 권한, 감사 로그를 관리한다.
대부분의 운영 데이터는 tournament_id를 가진다.
이 구조는 하나의 시스템에서 여러 대회를 관리할 수 있게 한다.

남자 리그전과 여자 리그전은 divisions 테이블에서 별도 부문으로 관리한다.
예를 들어 남자부는 code가 M이고 category가 male인 부문으로 만들 수 있다.
여자부는 code가 W이고 category가 female인 부문으로 만들 수 있다.
청소년부는 category가 youth인 부문으로 만들고 age_min과 age_max로 나이 범위를 지정할 수 있다.
오픈부, 혼성부, 사용자 정의 부문도 같은 구조로 확장할 수 있다.

선수 정보는 두 단계로 나눈다.
players 테이블은 선수의 기본 인물 정보를 저장한다.
tournament_players 테이블은 특정 대회와 특정 부문에 참가하는 정보를 저장한다.
시드 번호, 참가 번호, 소속 클럽, 대회별 표시 이름은 tournament_players에 저장한다.

경기 결과의 원본은 match_sets와 point_events이다.
match_sets는 세트별 최종 점수를 저장한다.
point_events는 라이브 득점 흐름을 저장한다.
matches는 공개 조회 성능을 위해 승자, 세트 승수, 총 득점 같은 결과 요약 값을 함께 저장한다.

순위는 언제든 재계산 가능해야 한다.
하지만 공개 화면이 빠르게 열리도록 마지막 계산 결과는 rankings_snapshots에 저장한다.

3. 테이블별 설계

3.1 tournaments

목적:
대회 하나의 기본 정보를 저장한다.

주요 컬럼:
id: UUID 기본키.
code: 공개 URL에 사용할 짧은 대회 코드.
name: 대회 이름.
location: 개최 장소.
start_date: 시작일.
end_date: 종료일.
timezone: 대회 시간대.
status: draft, published, running, finished, archived 중 하나.
default_language: 기본 언어 코드.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
code는 전체 시스템에서 unique이다.
end_date는 start_date보다 같거나 늦어야 한다.

설계 해석:
대회 등록 시 특정 날짜를 지정할 수 있다.
하루 대회는 start_date와 end_date를 같은 날짜로 입력한다.
여러 날 진행되는 대회는 start_date와 end_date를 다르게 입력한다.
실제 경기 배정 가능 시간은 schedule_blocks에서 날짜와 시간 단위로 더 자세히 지정한다.

인덱스:
code 조회를 위한 unique 인덱스.
status 조회를 위한 인덱스.
start_date 조회를 위한 인덱스.


3.2 divisions

목적:
남자부, 여자부, 오픈부, 주니어부 같은 대회 부문을 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
name: 부문 이름.
code: 부문 코드.
category: 부문 유형. male, female, mixed, youth, open, custom 중 하나.
age_min: 청소년부나 연령 제한 부문에서 사용할 최소 나이.
age_max: 청소년부나 연령 제한 부문에서 사용할 최대 나이.
sort_order: 표시 순서.
is_active: 사용 여부.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회 안에서 division code는 중복될 수 없다.
age_min과 age_max가 모두 있으면 age_max는 age_min보다 같거나 커야 한다.

인덱스:
tournament_id와 sort_order 기준 인덱스.
tournament_id와 category 기준 인덱스.

설계 해석:
남자 리그전과 여자 리그전은 각각 별도의 division으로 생성한다.
각 division은 별도의 stages, groups, matches, rankings_snapshots를 가진다.
따라서 남자부 리그전 순위와 여자부 리그전 순위는 서로 섞이지 않는다.
청소년부는 youth category와 age_min, age_max로 확장한다.


3.3 players

목적:
선수의 전역 인물 정보를 저장한다.

주요 컬럼:
id: UUID 기본키.
display_name: 화면 표시 이름.
family_name: 성.
given_name: 이름.
country_code: 국가 코드.
external_ref: 외부 시스템 선수 ID.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
display_name은 필수이다.
country_code는 선택 값이다.
external_ref는 외부 선수 DB 연동 시 사용한다.

인덱스:
display_name 검색 인덱스.
country_code 조회 인덱스.


3.4 tournament_players

목적:
특정 대회와 부문에 참가하는 선수 정보를 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
division_id: 부문 ID.
player_id: 선수 ID.
seed_no: 시드 번호.
entry_no: 참가 번호.
display_name_override: 대회에서 사용할 표시 이름.
club_name: 소속 클럽.
status: active, withdrawn, disqualified 중 하나.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회와 부문에서 같은 player_id는 한 번만 참가할 수 있다.
같은 대회와 부문에서 entry_no는 중복될 수 없다.
seed_no는 선택 값이다.

인덱스:
tournament_id와 division_id 기준 선수 목록 조회 인덱스.
player_id 기준 참가 이력 조회 인덱스.


3.5 stages

목적:
조별 예선, 본선 토너먼트, 순위 결정전 같은 대회 단계를 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
division_id: 부문 ID.
name: 단계 이름.
stage_type: round_robin, knockout, placement 중 하나.
default_match_duration_minutes: 해당 단계의 기본 경기 시간.
sort_order: 진행 순서.
status: draft, published, running, finished 중 하나.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 부문 안에서 sort_order는 단계 순서를 나타낸다.
default_match_duration_minutes는 0보다 커야 한다.

인덱스:
tournament_id, division_id, sort_order 기준 인덱스.

설계 해석:
리그전 단계는 stage_type을 round_robin으로 만든다.
토너먼트 단계는 stage_type을 knockout으로 만든다.
순위 결정전은 stage_type을 placement로 만든다.
기본 정책은 round_robin 30분, knockout 45분, placement 45분이다.
특정 대회나 특정 부문만 다른 시간이 필요하면 schedule_duration_rules에서 덮어쓴다.


3.6 groups

목적:
GROUP A, GROUP B, Quarter Final, Final 같은 경기 묶음을 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
code: 그룹 코드.
name: 그룹 표시 이름.
group_type: league, knockout, placement 중 하나.
sort_order: 표시 순서.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회와 부문 안에서 group code는 중복될 수 없다.

인덱스:
tournament_id, division_id, stage_id, sort_order 기준 인덱스.


3.7 group_members

목적:
선수가 어떤 그룹에 들어가는지 저장한다.

주요 컬럼:
id: UUID 기본키.
group_id: 그룹 ID.
tournament_player_id: 대회 참가 선수 ID.
slot_no: 그룹 안의 배정 번호.
source_rule: 이전 단계에서 올라온 규칙 설명.
created_at: 생성 시각.

중요 제약:
같은 그룹에 같은 tournament_player_id가 중복될 수 없다.
같은 그룹 안에서 slot_no는 중복될 수 없다.

인덱스:
group_id 기준 그룹 구성원 조회 인덱스.
tournament_player_id 기준 선수의 그룹 조회 인덱스.


3.8 courts

목적:
경기가 열리는 테이블 또는 코트를 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
name: 표시 이름.
sort_order: 표시 순서.
is_active: 사용 여부.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회 안에서 court name은 중복될 수 없다.

인덱스:
tournament_id와 sort_order 기준 인덱스.


3.9 officials

목적:
심판, 기록원, 운영 담당자를 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
name: 이름.
short_code: 짧은 표시 코드.
role_name: referee, scorer, organizer 같은 담당 역할.
is_active: 사용 여부.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회 안에서 short_code는 중복될 수 없다.

인덱스:
tournament_id와 role_name 기준 인덱스.


3.10 schedule_duration_rules

목적:
자동 경기 일정 생성에 사용할 경기 시간 규칙을 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID. null이면 전역 기본 규칙이다.
division_id: 부문 ID. null이면 대회 전체 규칙이다.
stage_type: round_robin, knockout, placement 중 하나.
match_duration_minutes: 경기 시간. 분 단위이다.
break_minutes: 경기 사이 휴식 시간. 분 단위이다.
is_default: 기본 규칙 여부.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
match_duration_minutes는 0보다 커야 한다.
break_minutes는 0 이상이어야 한다.
전역 기본 규칙은 stage_type별로 하나만 둘 수 있다.
대회 기본 규칙은 tournament_id와 stage_type별로 하나만 둘 수 있다.
부문 기본 규칙은 tournament_id, division_id, stage_type별로 하나만 둘 수 있다.

설계 해석:
기본 seed는 리그전 round_robin 30분, 토너먼트 knockout 45분, 순위 결정전 placement 45분을 제공한다.
특정 대회의 여자부 토너먼트만 50분으로 바꾸는 것도 가능하다.


3.11 schedule_official_rules

목적:
자동 경기 일정 생성 시 경기마다 심판 또는 운영 담당자를 몇 명 배정할지 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID. null이면 전역 기본 규칙이다.
division_id: 부문 ID. null이면 대회 전체 규칙이다.
stage_type: round_robin, knockout, placement 중 하나.
required_official_count: 자동 배정 시 반드시 배정할 심판 수. 1명 또는 2명이다.
max_official_count: 운영자가 배정할 수 있는 최대 심판 수. 1명 또는 2명이다.
is_default: 기본 규칙 여부.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
required_official_count는 1 또는 2이다.
max_official_count는 1 또는 2이다.
max_official_count는 required_official_count보다 같거나 커야 한다.
전역 기본 규칙은 stage_type별로 하나만 둘 수 있다.
대회 기본 규칙은 tournament_id와 stage_type별로 하나만 둘 수 있다.
부문 기본 규칙은 tournament_id, division_id, stage_type별로 하나만 둘 수 있다.

설계 해석:
기본 seed는 모든 단계에서 심판 1명을 자동 배정하고, 운영자가 필요하면 최대 2명까지 배정할 수 있게 한다.
특정 대회나 특정 부문에서 토너먼트 경기만 심판 2명을 필수로 지정할 수 있다.


3.12 schedule_blocks

목적:
자동 경기 일정 생성에 사용할 경기 가능 시간대를 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
division_id: 부문 ID. null이면 모든 부문에 적용할 수 있다.
stage_id: 단계 ID. null이면 모든 단계에 적용할 수 있다.
court_id: 경기장 또는 테이블 ID. null이면 모든 활성 경기장에 적용할 수 있다.
starts_at: 이 시간대의 시작 시각.
ends_at: 이 시간대의 종료 시각.
is_active: 사용 여부.
note: 운영 메모.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
ends_at은 starts_at보다 늦어야 한다.

설계 해석:
경기장 숫자는 courts 테이블의 활성 행 개수로 결정한다.
예를 들어 Table 1부터 Table 8까지 8개 행을 만들면 자동 스케줄러는 8개 경기장을 사용할 수 있다.
schedule_blocks는 각 경기장이 언제 사용 가능한지 알려준다.
court_id가 null인 schedule_block은 모든 활성 경기장에 공통 적용된다.
court_id가 지정된 schedule_block은 특정 경기장에만 적용된다.


3.13 matches

목적:
경기 하나의 일정, 상태, 결과 요약을 저장한다.

주요 컬럼:
id: UUID 기본키.
tournament_id: 대회 ID.
division_id: 부문 ID.
stage_id: 단계 ID.
group_id: 그룹 ID.
match_no: 대회 안에서 표시할 경기 번호.
scheduled_at: 경기 예정 시각.
duration_minutes: 해당 경기의 실제 배정 시간.
court_id: 코트 ID.
referee_id: 대표 심판 ID. 이전 호환성과 빠른 조회를 위한 선택 필드이다.
schedule_source: 일정 배정 출처. auto, manual, imported 중 하나이다.
schedule_locked: true이면 자동 스케줄러가 이 경기 일정을 덮어쓰지 않는다.
manual_note: 수동 변경 사유 또는 운영 메모.
status: scheduled, running, completed, cancelled, walkover 중 하나.
winner_tournament_player_id: 승자 ID.
player1_match_points: 선수 1 경기 승점.
player2_match_points: 선수 2 경기 승점.
player1_sets_won: 선수 1 세트 승수.
player2_sets_won: 선수 2 세트 승수.
player1_total_points: 선수 1 총 득점.
player2_total_points: 선수 2 총 득점.
version: 동시 수정 충돌 방지용 버전 번호.
confirmed_at: 결과 확정 시각.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
같은 대회 안에서 match_no는 중복될 수 없다.
status가 completed 또는 walkover이면 winner_tournament_player_id가 있어야 한다.
duration_minutes가 있으면 0보다 커야 한다.
점수와 세트 수는 0 이상이어야 한다.

인덱스:
tournament_id와 scheduled_at 기준 일정 조회 인덱스.
tournament_id와 status 기준 진행 상태 조회 인덱스.
group_id 기준 그룹 경기 조회 인덱스.
court_id와 scheduled_at 기준 코트별 일정 조회 인덱스.

설계 해석:
자동 스케줄러는 stage_type과 schedule_duration_rules를 보고 duration_minutes를 채운다.
리그전 경기는 기본적으로 30분이 들어간다.
토너먼트 경기는 기본적으로 45분이 들어간다.
운영자가 특정 경기만 수동으로 시간을 바꿀 수도 있다.
운영자가 특정 경기의 시간, 경기장, 대표 심판을 바꾸면 schedule_source를 manual로 저장한다.
schedule_locked가 true인 경기는 자동 재스케줄링 대상에서 제외한다.


3.14 match_players

목적:
한 경기에 참가하는 선수 또는 참가 슬롯을 저장한다.

주요 컬럼:
id: UUID 기본키.
match_id: 경기 ID.
tournament_player_id: 대회 참가 선수 ID.
side: player1 또는 player2.
source_slot: 토너먼트 자동 배정 설명.
assignment_source: 선수 배정 출처. auto, manual, imported 중 하나이다.
created_at: 생성 시각.
updated_at: 수정 시각.

중요 제약:
한 경기 안에서 같은 side는 하나만 존재할 수 있다.
한 경기 안에서 같은 tournament_player_id는 중복될 수 없다.

인덱스:
match_id 기준 경기 참가자 조회 인덱스.
tournament_player_id 기준 선수별 경기 조회 인덱스.

설계 해석:
자동 대진 생성으로 배정된 선수는 assignment_source를 auto로 저장한다.
운영자가 선수 스케줄 또는 대진을 수동 변경하면 assignment_source를 manual로 바꾼다.
수동 변경 이력은 audit_logs에 before_json과 after_json으로 저장한다.

4. 순위 계산 설계

기본 정렬 기준:
첫 번째 기준은 match_points이다.
두 번째 기준은 wins이다.
세 번째 기준은 set_difference이다.
네 번째 기준은 point_difference이다.
다섯 번째 기준은 points_for이다.
여섯 번째 기준은 head_to_head이다.
마지막 기준은 seed_no 또는 관리자 수동 결정이다.

DB 저장 원칙:
경기 결과 원본은 match_sets에 저장한다.
matches에는 계산된 요약 결과를 저장한다.
rankings_snapshots에는 공개 화면용 계산 결과를 저장한다.
순위 계산 로직은 백엔드 서비스에서 수행하는 것을 기본으로 한다.
DB 함수로 계산 로직을 만들 수 있지만, MVP에서는 애플리케이션 코드에서 계산하고 DB는 무결성과 저장을 담당한다.

동률 처리:
2명 동률이면 상호 전적을 우선한다.
3명 이상 동률이면 동률 선수 간 경기만 다시 계산한다.
동률 선수 간 부분 승점, 부분 세트 득실, 부분 점수 득실을 적용한다.
그래도 동률이면 seed_no 또는 관리자 수동 순위를 사용한다.
tie_break_note에 사람이 이해할 수 있는 설명을 저장한다.


5. 인덱스 전략

공개 대회 조회:
tournaments.code unique 인덱스를 사용한다.

선수 목록 조회:
tournament_players의 tournament_id, division_id, entry_no 인덱스를 사용한다.

일정 조회:
matches의 tournament_id, scheduled_at 인덱스를 사용한다.

열린 경기 조회:
matches의 tournament_id, status, scheduled_at 인덱스를 사용한다.

그룹 경기 조회:
matches의 group_id, match_no 인덱스를 사용한다.

선수별 일정 조회:
match_players의 tournament_player_id 인덱스와 matches 조인을 사용한다.

순위 조회:
rankings_snapshots의 tournament_id, division_id, stage_id, group_id, rank_no 인덱스를 사용한다.

감사 로그 조회:
audit_logs의 tournament_id, created_at 인덱스를 사용한다.


6. 자동 경기 일정 생성 설계

적용 여부:
현재 보강된 설계는 경기장 숫자에 맞춰 자동으로 경기 스케줄을 생성할 수 있는 구조를 가진다.
자동 스케줄러의 실제 알고리즘은 백엔드 서비스에서 구현한다.
데이터베이스는 스케줄러가 필요한 입력 데이터와 결과 저장 위치를 제공한다.

필요한 입력 데이터:
divisions에는 남자부, 여자부, 청소년부 같은 부문 정보가 있다.
stages에는 리그전, 토너먼트, 순위 결정전 같은 단계 정보가 있다.
groups와 group_members에는 각 조의 선수 배정 정보가 있다.
courts에는 사용할 경기장 또는 테이블 목록이 있다.
schedule_duration_rules에는 단계별 경기 시간이 있다.
schedule_official_rules에는 단계별 심판 배정 인원 규칙이 있다.
schedule_blocks에는 경기 가능한 시간대가 있다.
officials에는 대회에 참가하는 심판과 기록원 정보가 있다.
matches에는 생성된 경기와 배정된 경기장, 시작 시각, 경기 시간이 저장된다.
match_officials에는 각 경기에 배정된 심판 1명 또는 2명이 저장된다.

경기장 숫자 판단:
자동 스케줄러는 courts에서 tournament_id가 같고 is_active가 true인 행을 사용한다.
활성 경기장이 8개이면 같은 시간대에 최대 8경기까지 배정할 수 있다.
특정 시간대에 일부 경기장을 사용할 수 없으면 schedule_blocks에서 court_id별 시간대를 제한한다.

기본 경기 시간:
round_robin 단계는 기본 30분이다.
knockout 단계는 기본 45분이다.
placement 단계는 기본 45분이다.
이 값은 schedule_duration_rules에 저장한다.
특정 대회 또는 특정 부문에서 다른 시간이 필요하면 더 구체적인 규칙을 추가한다.
규칙 적용 우선순위는 부문별 규칙, 대회별 규칙, 전역 기본 규칙 순서이다.

자동 스케줄 생성 흐름:
먼저 부문과 단계를 선택한다.
다음으로 해당 단계의 그룹과 경기 목록을 만든다.
리그전이면 같은 그룹 안의 모든 선수 조합으로 경기를 만든다.
토너먼트이면 시드 또는 이전 단계 순위에 따라 대진 슬롯을 만든다.
스케줄러는 schedule_duration_rules에서 경기 시간을 찾는다.
스케줄러는 schedule_official_rules에서 심판 배정 인원을 찾는다.
스케줄러는 schedule_blocks에서 사용 가능한 날짜, 시간, 경기장을 찾는다.
스케줄러는 officials에서 사용 가능한 심판을 찾는다.
스케줄러는 경기 시간이 겹치지 않도록 scheduled_at, duration_minutes, court_id를 matches에 저장한다.
스케줄러는 심판 배정이 겹치지 않도록 match_officials에 1명 또는 2명을 저장한다.

자동 배정 충돌 방지 규칙:
같은 선수는 같은 시간대에 두 경기를 배정받을 수 없다.
같은 경기장은 같은 시간대에 두 경기를 배정받을 수 없다.
같은 심판은 같은 시간대에 두 경기에 배정받을 수 없다.
리그전 조별 경기는 가능한 한 같은 부문 안에서 시간대를 고르게 배정한다.
토너먼트 경기는 이전 라운드 종료 후 다음 라운드가 시작되도록 배정한다.

MVP에서 구현할 자동 스케줄러 기준:
남자부와 여자부를 별도 division으로 만들고 독립적으로 스케줄을 생성한다.
청소년부도 별도 division으로 만들면 같은 방식으로 스케줄을 생성한다.
리그전은 30분 단위로 배정한다.
토너먼트와 순위 결정전은 45분 단위로 배정한다.
경기장 수는 courts의 활성 데이터 수를 기준으로 한다.
운영자가 특정 경기의 시간과 경기장을 수동 수정할 수 있게 한다.
운영자가 특정 경기의 심판 1명 또는 2명을 수동 수정할 수 있게 한다.
운영자가 선수 대진 또는 선수 일정을 수동 수정할 수 있게 한다.
수동 수정된 경기는 schedule_source 또는 assignment_source를 manual로 저장하고 audit_logs에 변경 이력을 남긴다.


7. 운영과 보안 설계

비밀번호:
users.password_hash에는 평문 비밀번호를 저장하지 않는다.
애플리케이션에서 bcrypt, argon2 같은 안전한 해시를 사용한다.

권한:
roles와 user_roles로 권한을 관리한다.
tournament_id가 null인 user_roles는 시스템 전체 권한을 의미한다.
tournament_id가 있는 user_roles는 특정 대회 권한을 의미한다.

감사 로그:
점수 변경, 결과 확정, 경기 취소, 선수 기권, 순위 재계산은 audit_logs에 저장한다.
before_json과 after_json은 jsonb로 저장한다.

동시 수정:
matches.version을 사용한다.
클라이언트가 저장 요청을 보낼 때 현재 version을 함께 보낸다.
서버의 version과 다르면 저장을 거부하고 최신 정보를 다시 읽도록 한다.


8. SQL 파일 구성

00_create_database.sql:
showdown_tournament 데이터베이스를 생성한다.
psql에서 실행하는 것을 기준으로 한다.

V1__create_schema.sql:
확장 기능, enum 타입, 테이블, 제약 조건, 인덱스를 생성한다.
showdown_tournament 데이터베이스에 접속한 뒤 실행한다.

V2__seed_minimum_data.sql:
기본 권한 역할과 기본 순위 규칙 템플릿을 입력한다.
실제 대회, 실제 선수, 실제 경기 샘플 데이터는 넣지 않는다.


9. 구현 시 주의사항

외래키 순서:
상위 테이블을 먼저 만들고 하위 테이블을 나중에 만든다.
삭제는 운영 정책상 물리 삭제보다 status 또는 is_active 변경을 우선한다.

마이그레이션:
실제 애플리케이션에서는 Prisma, TypeORM, Knex, Flyway 같은 마이그레이션 도구를 사용할 수 있다.
하지만 이 SQL 스크립트는 특정 ORM에 종속되지 않게 작성한다.

백업:
대회 진행 중에는 점수 입력 데이터가 매우 중요하다.
PostgreSQL 백업과 audit_logs 보존 정책을 반드시 운영 계획에 포함한다.

확장:
라이브 티커는 point_events를 기반으로 만든다.
즐겨찾기는 favorites를 기반으로 만든다.
여러 대회와 여러 부문은 tournaments, divisions, stages 구조로 지원한다.
